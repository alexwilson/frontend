# feed-validator-action

A GitHub Action that runs the W3C feedvalidator over an Atom or RSS feed in CI. The validator is bundled as a git submodule at `vendor/feedvalidator`, pinned to a specific upstream SHA. Dependabot keeps it current and the contract tests catch behaviour changes before they reach your CI.

## Usage

```yaml
- uses: actions/checkout@v5
- uses: alexwilson/feed-validator-action@v1
  with:
    feed: public/feed.xml
    origin-url: https://example.com/feed.xml
```

You don't need `submodules: true` on your own checkout. The vendored feedvalidator ships inside the published action.

## Inputs

| Name | Default | Notes |
|---|---|---|
| `feed` | (required) | Path or URL of the feed. Local paths resolve relative to the workspace; URLs are fetched live. |
| `origin-url` | — | For local files: the URL this feed will be served from. The validator treats the file as if loaded there, so the self-link check passes for an artefact that hasn't shipped yet. Ignored when `feed` is a URL. |
| `python-version` | `3.x` | Override if a future Python rev breaks the validator. |
| `fail-on` | `errors` | `errors` (any validator error), `warnings` (any warning or error), or `never` (report only, never fail the step). |
| `compatibility` | `AA` | Validator strictness: `A` (loosest), `AA` (mimics the online validator), or `AAA` (experimental; upstream warns rules may change). |

## Outputs

| Name | Notes |
|---|---|
| `errors` | Number of errors reported. |
| `warnings` | Number of warnings reported. |
| `issues` | Errors + warnings (info events are excluded). |
| `report-path` | Path to a text file with the full report, info events included. Pair with `actions/upload-artifact` for post-mortem inspection. |

## Examples

Validate a built feed against the URL it will live at:

```yaml
- uses: alexwilson/feed-validator-action@v1
  with:
    feed: public/feed.xml
    origin-url: https://example.com/feed.xml
```

Validate a live feed, never fail, save the report as an artifact:

```yaml
- id: feed
  uses: alexwilson/feed-validator-action@v1
  with:
    feed: https://example.com/feed.xml
    fail-on: never
- uses: actions/upload-artifact@v5
  with:
    name: feed-validator-report
    path: ${{ steps.feed.outputs.report-path }}
```

## Versioning

Tags follow semver. Pin to a major (`@v1`) for automatic patch/minor updates; pin to a full version (`@v1.2.3`) to freeze. Vendored feedvalidator bumps ship as patches.

## Dependency pinning

Runtime dependencies are locked in `requirements.lock`, generated by `pip-compile --generate-hashes` from `requirements.in`. The action installs with `--require-hashes`, so a swapped PyPI artifact gets rejected even if the version number matches. The lockfile is regenerated automatically when Dependabot bumps the submodule.

## Development

```sh
python3.12 -m venv .venv
.venv/bin/pip install --require-hashes -r requirements.lock
.venv/bin/pytest tests/
```

The tests cover both directions of the contract: valid fixtures shouldn't report errors, invalid ones should. They also pin the exit-code logic for `fail-on`. If a submodule bump changes how the validator classifies an issue, expect to see it here first.

## Bumping the vendored validator manually

Dependabot handles this on a weekly schedule. By hand:

```sh
git -C vendor/feedvalidator fetch
git -C vendor/feedvalidator checkout <new-sha>
.venv/bin/pip install pip-tools
.venv/bin/pip-compile --generate-hashes --output-file requirements.lock requirements.in
.venv/bin/pip install --require-hashes -r requirements.lock
.venv/bin/pytest tests/
git add vendor/feedvalidator requirements.lock
```
